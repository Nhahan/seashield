## 1. 크로스플랫폼 비동기 I/O 추상화 — kqueue/epoll Reactor 직접 구현

**무엇을 보여주나**
서드파티(asio, libuv) 없이 kqueue(macOS)·epoll(Linux)을 직접 다루는 Reactor 이벤트 루프를 구현하고, 두 OS의 차이를 흡수하는 추상화 계층을 설계. non-blocking 소켓, 수평 트리거 처리, 송신 버퍼 backpressure까지 포함.

**왜 5년차다운가**
라이브러리 사용자가 아니라 **라이브러리를 만들 수 있는 쪽**임을 보인다. 특히 "추상화 경계를 어디에 둘 것인가"(readiness 기반 두 API는 묶고, completion 기반 IOCP는 인터페이스 레벨을 올려 수용)는 API 설계 경험이 있어야 나오는 판단.

**예상 면접질문 (꼬리질문 포함)**
- epoll과 kqueue의 API 차이는? 추상화하면서 가장 어려웠던 불일치는 무엇이었나?
- 레벨 트리거(LT)와 엣지 트리거(ET) 중 무엇을 썼고 왜인가? ET로 바꾸면 무엇이 달라지나?
- IOCP는 왜 같은 추상화에 바로 안 들어가나? (→ readiness vs completion, Reactor vs Proactor)
- `select`/`poll` 대비 epoll이 빠른 이유는? O(n)이 어디서 사라지나?
- 송신 버퍼가 가득 차면(EWOULDBLOCK) 어떻게 처리했나? 느린 클라이언트가 서버를 막을 수 있나?
- asio를 안 쓴 이유는? 실무라면 어떤 선택을 하겠나? (→ "학습/증명 목적엔 직접 구현, 실무 신규 프로젝트라면 asio 검토"가 균형 잡힌 답)

**방산 도메인 연결**
무기체계 소프트웨어는 임의의 서드파티 도입이 어렵다(공급망·인증 제약). 표준 라이브러리만으로 OS 위에 직접 구축하는 능력과 이식성 있는 추상화 설계는 방산 개발 환경에 직결되는 역량.

**증빙**: `net/event_loop.h`, `net/kqueue_event_loop.cpp`, `net/epoll_event_loop.cpp` · `docs/architecture/network-design.md` (대안 비교 포함) · CI 매트릭스(macOS/Linux 동시 통과)

## 2. TCP/UDP 하이브리드 프로토콜 + 커스텀 Reliable UDP

**무엇을 보여주나**
데이터 특성(빈도·신뢰성 요구·최신성)에 따라 TCP와 UDP에 역할을 분리하고, UDP 위에 시퀀스/ACK 비트필드 기반 **선택적 신뢰성 레이어**를 직접 구현. 자체 바이너리 직렬화와 스냅샷+델타 압축까지 프로토콜 전체를 설계. 여기에 **멀티클라이언트 서버 메커니즘**(TCP 길이 프리픽스 프레이밍, 클라이언트별 송신 큐·backpressure·느린 클라이언트 격리, 연결 수명주기 상태머신 — 기획서 §4.8)을 더해 "여러 클라이언트를 받아 처리하는 서버"라는 **하드 요구사항을 직접 증명**한다.

**왜 5년차다운가**
"UDP가 빠르다더라" 수준이 아니라, head-of-line blocking·재전송 무가치성 같은 **원리에서 채널 설계를 도출**하고, TCP를 재구현하지 않고 필요한 신뢰성만 만드는 절제를 보인다. 요구사항("UDP/TCP 기반 게임서버")을 가장 정면으로 증명하는 항목.

**예상 면접질문 (꼬리질문 포함)**
- 실시간 게임에서 TCP가 부적합한 이유를 구체적으로? (→ HOL blocking, 재전송된 과거 상태의 무가치, Nagle/지연 ACK)
- TCP는 바이트 스트림인데 메시지 경계는 어떻게 구분하나? 한 번의 read에 메시지 1.5개가 오면? (→ 길이 프리픽스 프레이밍 + 세션별 수신 버퍼에 부분 수신 누적, 상태 보존 파서)
- 느린 클라이언트 하나가 다른 클라이언트나 서버 틱을 막을 수 있나? (→ 클라이언트별 송신 큐 상한 + 초과 시 해당 세션만 절단하는 격리 정책, 부하 테스트로 실증)
- ACK 비트필드 32비트로 충분한가? 33개 이상 연속 유실되면? (→ 재전송 타임아웃 경로, 채널 특성상 허용)
- 시퀀스 번호 wrap-around는 어떻게 안전하게 비교하나?
- Reliable-Ordered 채널은 왜 안 만들었나? (→ 순서까지 필요하면 그것이 곧 TCP, 해당 데이터는 TCP로 보냄)
- UDP 패킷이 TCP 세션과 같은 클라이언트의 것인지 어떻게 식별·인증하나? (→ 세션 토큰 바인딩)
- 델타 압축에서 기준 스냅샷을 클라이언트가 잃어버리면? (→ ACK 기반 기준 관리, full snapshot 폴백)
- MTU와 단편화는 고려했나? (→ 페이로드 상한 ~1200B 정책 + 스냅샷은 독립 디코딩 가능한 배치 단위로 분할, IP 단편화에 맡기지 않는 이유까지)
- LAN이라도 UDP 스푸핑은 어떻게 막나? (→ 세션 토큰 바인딩, 단 암호화/HMAC는 LAN 신뢰 전제로 범위 외 선언 — 위협 표면을 알고 배제했음을 답변)

**방산 도메인 연결**
전술데이터링크(Link-16 등)·함내 체계 간 통신도 결국 "신뢰성 등급별 메시지 채널 설계" 문제. 대역폭 제약 환경에서의 양자화·우선순위 설계 경험은 그대로 전이된다.

**증빙**: `protocol/` 전체 · `docs/architecture/protocol-spec.md` · 패킷로스 주입 테스트 결과(§6 연계) · 대역폭 측정(성능 보고서)

## 3. 서버 권위 · 고정 틱 · 결정론 시뮬레이션 + 리플레이/AAR

**무엇을 보여주나**
60Hz 고정 타임스텝, 갱신 순서 고정, 시드 고정 PRNG, 입력 저널링으로 **동일 입력 → 동일 결과**를 보장하는 시뮬레이션 코어. 그 위에 수십 KB짜리 저널만으로 전체 교전을 재현하는 리플레이와 AAR(사후분석)을 구축.

**왜 5년차다운가**
결정론은 "나중에 추가"가 불가능한 아키텍처 속성임을 알고 **P2에서 회귀 테스트부터 구축**하는 순서 감각. 부동소수점 결정론의 한계(크로스 플랫폼 비트 동일성 비목표)를 인지하고 경계를 명시하는 것도 성숙도의 증거.

**예상 면접질문 (꼬리질문 포함)**
- 결정론을 깨뜨리는 요인들을 아는 대로 말해보라. (→ 가변 dt, 순회 순서, 비고정 시드, 벽시계 의존, FP 환경 차이, 미초기화 메모리)
- 부동소수점 연산은 플랫폼 간 결과가 같은가? 동일 바이너리면 항상 같은가? (→ 플랫폼 간: FMA, 컴파일러 최적화, x87/SSE로 다를 수 있음. 동일 바이너리라도 libm 구현 차·FMA contraction 여지가 있어 `-ffp-contract=off` + 초월함수 고정 구현으로 통제)
- 리플레이가 어느 날 안 맞는다면 어떻게 디버깅하나? (→ 틱별 상태 해시 이분 탐색으로 첫 발산 틱 특정)
- 고정 틱인데 처리시간이 16.6ms를 넘으면? (→ 지터 측정, 데스 스파이럴 방지 정책: 최대 캐치업 틱 수 제한)
- 서버 권위 모델에서 클라이언트 지연은 어떻게 보상하나? (→ 보간 버퍼; 예측을 안 쓴 도메인 근거까지)

**방산 도메인 연결**
훈련 시뮬레이터·전투체계의 핵심 요구가 바로 **재현성**이다. 같은 시나리오에서 같은 결과가 나와야 검증·인증·사후분석이 가능하다. AAR은 실제 훈련체계의 표준 구성요소.

**증빙**: `sim/`, `replay/` · 결정론 회귀 테스트(golden 해시) · 리플레이 시연 영상 · AAR 리포트 샘플

## 4. 멀티스레딩 설계 — 시뮬레이션/I/O 분리, Lock-free 큐

**무엇을 보여주나**
I/O 스레드(Reactor)와 시뮬레이션 스레드(고정 틱)를 분리하고, 둘 사이를 lock-free SPSC 큐 한 쌍으로만 연결해 **공유 가변 상태를 큐 경계로 격리**한 메시지 전달 설계 — 레이스 가능 지점을 "검증 가능한 한 곳"으로 좁히고, 그 한 곳을 단위 테스트 + CI 상시 TSan으로 기계적으로 검증.

**왜 5년차다운가**
"스레드를 많이 쓰는 것"이 아니라 **목표 규모(≤8 클라, ≤500 엔티티)에 맞는 최소 스레드 수를 근거와 함께 선택**하고, 확장 경로를 문서로 남기는 것. lock-free를 직접 구현하면서 memory ordering을 다뤄본 경험은 동시성 질문 전반의 방어선이 된다.

**예상 면접질문 (꼬리질문 포함)**
- 왜 스레드가 2개뿐인가? 더 못 늘리나? (→ 규모 산정 근거 + 확장 경로: 시스템 병렬화, SO_REUSEPORT)
- SPSC 큐에서 acquire/release가 각각 어디에 왜 필요한가? seq_cst와의 차이는?
- ABA 문제가 무엇이고 이 설계에서 왜 발생하지 않나? (→ SPSC ring buffer는 생산자/소비자가 각자 단조 증가 인덱스를 단독 소유해 CAS 경쟁 자체가 없으므로 비해당. MPSC로 확장하면 달라지므로 그때는 Vyukov 시퀀스 카운터 방식으로 슬롯 세대를 구분)
- I/O 스레드가 1개인데 수신 큐가 SPSC면 다중 I/O 스레드 확장 시에는? (→ SO_REUSEPORT 증설 시 MPSC 전환 — 확장 경로를 설계 문서에 선반영)
- 큐가 가득 차면? (→ backpressure 정책: 스냅샷은 drop-oldest, 명령은 차단 불가이므로 용량 설계)
- mutex 기반과 lock-free의 실측 차이를 재봤나? (→ 벤치마크로 답 — "측정해보니 이 규모에선 차이 미미, 그러나 tail latency에서 이점" 같은 정직한 답이 강함)
- TSan이 잡아준 실제 버그가 있었나? (→ 개발 중 사례 기록해둘 것)

**방산 도메인 연결**
실시간 체계의 멀티스레딩은 "빠름"보다 **예측 가능성**(지터 최소화, 우선순위 역전 방지)이 중요. 공유 상태 제거 설계는 안전성 분석(어디서 레이스가 불가능한가의 논증)이 쉬워 방산 검증 문화와 정합.

**증빙**: `core/spsc_queue.h` + 단위 테스트 (MPSC는 I/O 스레드 증설 시 확장) · 스레딩 설계 문서 · TSan CI 잡 로그 · 틱 지터 측정(성능 보고서)

## 5. 도메인 알고리즘 — 칼만 필터 추적 · 비례항법유도(PN)

**무엇을 보여주나**
레이더 노이즈 측정(플롯)으로부터 칼만 필터(CV 모델)로 트랙을 추정하고, 그 **추정 트랙**(진실이 아닌)을 데이터링크 삼아 PN 유도(a_cmd = N·V_c·λ̇)로 요격하는 **센서→추적→유도의 인과 사슬 전체**를 구현. 수식 유도 과정과 레퍼런스(NumPy) 대조 검증을 문서화.

**왜 5년차다운가**
공식을 베끼는 게 아니라, Q/R 튜닝이 트랙 품질을 거쳐 miss distance로 전파되는 **시스템 수준의 인과**를 설계·실험한다. "필터를 진실 데이터에 물리면 의미가 없다"는 지점을 아는 것이 시뮬레이션 엔지니어링의 분기점.

**예상 면접질문 (꼬리질문 포함)**
- 칼만 필터의 predict/update 두 단계를 수식 수준으로 설명해보라. 칼만 이득이 의미하는 바는?
- Q를 키우면/줄이면 무슨 일이 생기나? 기동 표적에서 CV 모델의 한계는? (→ 기동 시 추정 지연·발산, IMM이 다음 단계인 이유)
- 게이팅에 마할라노비스 거리를 쓰는 이유는? 유클리드 거리와 뭐가 다른가?
- PN에서 항법상수 N의 물리적 의미는? N이 너무 크면/작으면? (→ 노이즈 증폭 vs 추적 지연, 통상 3~5)
- 시선각속도(λ̇)는 어떻게 계산하나? (→ 원시 측정의 유한차분이 아니라 필터의 위치·속도 추정으로 ω = (r × v_rel)/|r|² — 노이즈 증폭 회피) 그래도 남는 노이즈는 가속 명령에 어떤 영향을 주나?
- 요격률을 올리고 싶다. 필터를 좋게 하는 것과 N을 조정하는 것 중 무엇이 효과적이었나? (→ 실험 보고서로 답)

**방산 도메인 연결**
칼만 필터·PN은 유도무기/전투체계 직무의 **사실상 공용어**. 게임서버 출신이 이 언어를 구사하면 "도메인 전환 비용이 낮은 지원자"로 읽힌다. 면접관(현업 엔지니어)과 가장 깊은 대화가 가능한 항목.

**증빙**: `sim/tracking/`, `sim/guidance/` · `docs/architecture/simulation-models.md` (수식 유도) · `reports/guidance-experiments.md` (N·Q/R 스윕 실험) · 단위 테스트(레퍼런스 대조)

## 6. 테스트·검증 문화 — 결정론 회귀, 불량환경 주입, CI 매트릭스

**무엇을 보여주나**
단위/통합 테스트 위에 ① 틱별 상태 해시 golden 비교(결정론 회귀), ② 자체 UDP 프록시로 패킷로스·지연·재정렬 주입(프로토콜 강건성), ③ macOS/Linux × ASan/UBSan/TSan CI 매트릭스 — **3중 검증 체계**.

**왜 5년차다운가**
테스트를 "기능 확인"이 아니라 **아키텍처 속성(결정론, 프로토콜 강건성, 스레드 안전성)의 지속적 증명 장치**로 쓴다. 특히 불량환경 주입 툴을 직접 만들어 재현 가능한 네트워크 테스트를 구축하는 것은 실무에서 장애를 겪어본 연차의 발상.

**예상 면접질문 (꼬리질문 포함)**
- 시뮬레이션처럼 출력이 복잡한 시스템은 무엇을 assert하나? (→ 속성 기반: 해시 일치, 보존량, 물리 불변식)
- golden 해시가 깨졌는데 의도된 변경인지 버그인지 어떻게 구분하나? (→ 리뷰 절차 + 발산 틱 이분탐색)
- 패킷로스 테스트를 OS 도구(tc, dummynet) 대신 자체 프록시로 한 이유는? (→ 시드 고정 재현성, CI 이식성)
- 플레이키(flaky) 테스트가 생기면 어떻게 다루나?
- 커버리지는 어느 수준이며, 숫자보다 중요하게 본 것은?

**방산 도메인 연결**
방산 SW 개발의 본질은 **V&V(Verification & Validation)**. "요구→테스트"가 닫혀 있는 포트폴리오는 그 자체로 방산 프로세스(V-model, 시험평가) 적응력의 증거가 된다.

**증빙**: `tests/` 트리 · 결정론 golden 파일과 갱신 이력 · `tools/netem_proxy/` · GitHub Actions 워크플로우 + 배지

## 7. 측정 기반 성능 엔지니어링

**무엇을 보여주나**
"빠르게 만들었다"가 아니라 **측정 → 병목 식별 → 개선 → 재측정**의 사이클을 보고서로 남긴다: 틱 처리시간 분포(p50/p99), 틱 지터, 클라이언트 수·엔티티 수 스케일링 곡선, 스냅샷 대역폭(전체 vs 델타 압축), 부하 한계점.

**왜 5년차다운가**
p99·지터 같은 **꼬리 지연(tail latency)** 중심 사고, 그리고 "목표 수치를 먼저 정하고(틱 예산 16.6ms의 50%) 그 안에서 멈추는" 최적화 절제. 추측이 아닌 프로파일러(Instruments, perf)와 자체 계측으로 말한다.

**예상 면접질문 (꼬리질문 포함)**
- 평균이 아니라 p99를 보는 이유는?
- 가장 큰 병목은 무엇이었고 어떻게 찾았나? (→ 실제 사례를 보고서에서 인용)
- 델타 압축으로 대역폭이 얼마나 줄었나? 그 대가는? (→ 서버 메모리·복잡도, ACK 의존)
- 클라이언트당 대역폭 산정 근거를 말해보라. (→ 엔티티 ~20B × 100 × 30Hz ≈ 480kbps 풀 스냅샷 → 델타 후 <256kbps 목표 — 기획서 §6의 산수를 그대로, 실측치로 마감)
- 메모리 할당 전략은? 틱 루프 안에서 heap 할당을 피했나? (→ 풀링/사전 할당, 측정 근거)
- 클라이언트를 8 → 64로 늘리면 어디가 먼저 무너질 것 같나? (→ 스케일링 곡선 기반 추론)

**방산 도메인 연결**
실시간 체계는 "평균적으로 빠름"이 아니라 **마감시간(deadline) 내 완료 보장**이 요구된다. 지터·최악 실행시간(WCET) 관점의 측정 습관은 방산 실시간 SW와 직접 연결.

**증빙**: `reports/performance-report.md` (그래프 포함) · 계측 코드(`core/metrics.h`) · 부하 테스트 드라이버(`tools/loadgen/`)

## 8. 요구사항 추적성 — V-model 친화적 문서 체계

**무엇을 보여주나**
"기획서의 요구 → 설계 문서의 결정 → 코드 모듈 → 검증 테스트"를 잇는 **추적성 매트릭스**. 모든 아키텍처 결정에 trade-off와 기각된 대안을 기록(경량 ADR 스타일).

**왜 5년차다운가**
코드가 아니라 **프로세스를 설계**한다. 혼자 하는 포트폴리오에도 추적성을 적용했다는 것은 "팀·조직 단위 개발 문화를 이해하고 자기 규율로 내재화했다"는 신호.

**예상 면접질문 (꼬리질문 포함)**
- V-model을 게임서버 개발 경험과 어떻게 접목하겠나? 애자일과 충돌하지 않나?
- 설계 변경이 생기면 추적성 매트릭스는 어떻게 유지하나? (→ 변경 비용을 줄이는 문서 granularity 선택)
- 기각한 대안 중 가장 아까웠던 것은? (→ ADR에서 인용)

**방산 도메인 연결**
방산 SW는 개발 표준(무기체계 SW 개발 매뉴얼 등)상 산출물·추적성이 의무다. 이를 **요구받기 전에 스스로 하는 지원자**라는 포지셔닝 — 발표에서 "방산 개발 프로세스를 조사하고 맞춰 준비했다"고 명시적으로 언급할 것.

**증빙**: `docs/traceability-matrix.md` · 각 설계 문서의 "대안과 기각 사유" 섹션 · 본 문서 체계 자체

## 9. "멀티클라이언트 = CIC 운용석" — 요구사항의 도메인적 재해석

**무엇을 보여주나**
"온라인일 필요 없는 멀티클라이언트 UDP/TCP 서버"라는 다소 기술 중심적인 요구사항을, **함정 전투정보실(CIC)의 다중 운용 콘솔 구조**라는 도메인 시스템으로 번역해 기획 전체의 뼈대로 삼은 것 (기획서 §3.1).

**왜 5년차다운가**
주니어는 요구사항을 구현하고, 시니어는 요구사항의 **의도**를 찾아 더 나은 문제 정의로 바꾼다. 기술 데모가 아니라 "왜 이 아키텍처가 이 도메인에서 필연인가"를 말할 수 있는 것이 기획력·커뮤니케이션 역량의 증거.

**예상 면접질문 (꼬리질문 포함)**
- 왜 굳이 운용석 3종을 나눴나? 1인 플레이로 충분하지 않나? (→ 멀티클라이언트 요구의 도메인 정당화 + 역할별 권한이라는 설계 과제 확보)
- 실제 전투체계 콘솔 구조를 조사해봤나? 어디까지가 공개 정보인가? (→ 공개자료 경계를 지킨 점 자체가 보안 의식 어필)
- 이 구조라면 다음에 무엇을 붙이고 싶나? (→ 확장 백로그: CIWS 계층방어, 다함정 협동교전 등)

**방산 도메인 연결**
이 항목이 발표의 **오프닝**이다. "게임서버 기술 스택"과 "방산 도메인"이 별개가 아니라 구조적으로 같은 문제임을 보여주며 시작하면, 이후 모든 기술 항목이 도메인 이야기로 들린다.

**증빙**: 기획서 §1.3(요구사항 매핑), §3.1(재해석 논리) · 발표 스토리라인 문서(P6)
